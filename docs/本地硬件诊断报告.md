## 三、卡顿原因分析

### 3.1 主要原因：内存严重不足

**当前状态**：
- 可用内存仅 **754 MB**
- 虚拟内存使用高达 23.5 GB（严重依赖 pagefile）
- **这是造成严重卡顿的根本原因**

### 3.2 卡顿链分析

```
内存不足 → 系统使用虚拟内存(pagefile) → 频繁读写磁盘
       → Next.js 编译需要大量内存 → 内存不足
       → 页面切换触发重新编译 → 卡顿加剧
```

### 3.3 Next.js 开发模式特点

从日志可以看到页面切换时的编译情况：
```
GET /account/model 304 in 34.4s (compile: 33.6s)
○ Compiling /account/model ...
```

**编译耗时 33.6 秒** 是因为：
1. 内存不足导致编译变慢
2. 同时 Docker 容器占用大量内存
3. 系统频繁使用 swap 导致 I/O 阻塞

## 四、问题诊断结论

### 4.1 诊断结果

| 问题 | 严重程度 | 说明 |
|------|----------|------|
| **内存严重不足** | 🔴 严重 | 8GB 勉强达标，当前已耗尽 |
| **虚拟内存过度使用** | 🔴 严重 | 23GB 虚拟内存使用，频繁 swap |
| **Node.js 内存占用高** | 🟡 中等 | 1.5GB，可能是编译缓存 |
| **CPU 单核心** | 🟡 中等 | 编译只能使用单核 |

### 4.2 为什么卡顿

1. **物理内存耗尽**：754MB 可用内存远低于 Next.js 编译需求
2. **虚拟内存瓶颈**：频繁读写 pagefile.sys（机械硬盘/慢速SSD）
3. **编译内存不足**：Next.js 编译需要 2-4GB 内存，当前无法满足
4. **并发请求**：多个 API 请求同时触发编译，内存竞争

## 五、优化建议

### 5.1 立即可执行的优化

| 优化项 | 操作 | 预期效果 |
|--------|------|----------|

| 容器 | 内存 | 关闭建议 |

| fastgpt-minio | 144 MB | 🟢 保留 |

## 六、临时解决方案（不升级硬件）

### 6.1 方案一：精简 Docker + 关闭其他应用

1. 关闭所有非必要应用（浏览器、IDE、微信等）
2. 只保留核心容器：pg, mongo, redis
3. 保持 Next.js 运行

**预计可腾出**: 2-3 GB 内存

### 6.2 方案二：预热编译

首次访问页面后，Next.js 会缓存编译结果。预热步骤：

1. 访问 http://localhost:3000
2. 等待编译完成（看终端日志）
3. 再访问其他页面（使用缓存，加快速度）

### 6.3 方案三：使用生产模式测试

```bash
cd projects/app
pnpm build
pnpm start
```

生产模式只会编译一次，运行更快。
