## 一、硬件需求概述

### 1.1 开发模式 vs 生产模式

FastGPT 开发环境需要同时运行：
1. **Docker 容器服务**：数据库、缓存、向量库等
2. **Node.js 开发服务器**：Next.js 前端 + 后端服务

### 1.2 最低硬件配置

| 组件 | 最低配置 | 推荐配置 |
|------|----------|----------|
| **CPU** | 4 核 | 8 核+ |
| **内存** | 8 GB | 16 GB+ |
| **硬盘** | 40 GB SSD | 100 GB SSD |
| **操作系统** | Windows 10/11 或 macOS | Windows 11 或 macOS |

## 三、Node.js 开发服务器资源

### 3.1 Next.js 开发模式特点

Next.js 开发模式会：
- 保留 webpack/rspack 编译缓存
- 保持开发服务器运行
- 监听文件变化进行热更新

### 3.2 内存需求分析

```typescript
// 项目中发现的配置
// packages/service/common/system/constants.ts
export const serviceRequestMaxContentLength =
  Number(process.env.SERVICE_REQUEST_MAX_CONTENT_LENGTH || 10) * 1024 * 1024; // 10MB
```

| 操作 | 内存峰值 | 说明 |
|------|----------|------|
| 首次编译 | 2-3 GB | 编译所有页面和 API |
| 热更新 | 500MB-1GB | 仅编译变更部分 |
| API 请求 | 200-500MB | 根据请求大小变化 |

### 3.3 开发模式内存占用

| 阶段 | 内存占用 | 持续时间 |
|------|----------|----------|
| 启动 | 2-3 GB | 1-2 分钟 |
| 稳定运行 | 1.5-2 GB | 持续 |
| 大文件上传/处理 | 可能到 4 GB | 临时 |

## 四、数据库资源需求

### 4.1 PostgreSQL (pgvector)

```yaml
# docker-compose.yml 配置
environment:
  - POSTGRES_USER=username
  - POSTGRES_PASSWORD=password
volumes:
  - ./pg/data:/var/lib/postgresql/data
```

| 资源 | 最低 | 推荐 |
|------|------|------|
| 内存 | 1 GB | 2 GB |
| 磁盘 | 5 GB | 20 GB |
| CPU | 1 核 | 2 核 |

**向量库特殊性**：pgvector 用于存储和检索向量嵌入，需要内存足够以保持索引在内存中。

### 4.2 MongoDB

```yaml
# docker-compose.yml 配置
command: mongod --keyFile /data/mongodb.key --replSet rs0
environment:
  - MONGO_INITDB_ROOT_USERNAME=myusername
  - MONGO_INITDB_ROOT_PASSWORD=mypassword
volumes:
  - ./mongo/data:/data/db
```

| 资源 | 最低 | 推荐 |
|------|------|------|
| 内存 | 1 GB | 2 GB |
| 磁盘 | 5 GB | 20 GB |
| CPU | 1 核 | 1 核 |

### 4.3 Redis

## 七、不同开发场景配置

### 7.1 轻量级开发（4GB 内存电脑）

如果内存有限，可以：

1. **减少 Docker 容器**：
   - 关闭 `aiproxy`（如不需要 AI 代理功能）
   - 关闭 `fastgpt-plugin`（如不需要插件）
   - 使用外部 AI API 代替

2. **限制容器内存**：

```yaml
# 在 docker-compose.yml 中添加
services:
  pg:
    deploy:
      resources:
        limits:
          memory: 512M
  mongo:
    deploy:
      resources:
        limits:
          memory: 512M
```

## 八、性能优化建议

### 8.1 Docker 性能优化

1. **使用 SSD**：数据库对磁盘 I/O 敏感
2. **分配足够内存**：避免 swap
3. **配置 Docker 资源限制**

```json
{
  "resources": {
    "limits": {
      "memory": "8g",
      "cpus": "4"
    }
  }
}
```

### 8.2 Next.js 开发优化

1. **使用 `pnpm` 而不是 `npm`**：更快、更省内存
2. **及时清理 `.next` 缓存**：如果遇到奇怪的编译问题
3. **避免一次打开太多页面**：每个页面都会编译

### 8.3 数据库优化

1. **为 PostgreSQL 分配足够内存**：用于缓存向量索引
2. **使用本地 Docker**：避免网络存储的性能损失

## 十、总结

### 10.1 快速对照表

| 场景 | 最低配置 | 推荐配置 |
|------|----------|----------|
| **CPU** | 4 核 | 8 核+ |
| **内存** | 8 GB | 16 GB |
| **硬盘** | 40 GB SSD | 100 GB+ SSD |
| **能否开发** | 能，但卡顿 | 流畅 |

### 10.2 检查方法

打开任务管理器，检查：
1. **Docker Desktop** 内存使用应 < 8 GB
2. **Node.js** 进程内存使用通常 1-3 GB
3. **可用内存** 应始终保持 > 2 GB

### 10.3 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 编译很慢 | 内存不足 | 增加内存或关闭其他应用 |
| 数据库连接失败 | 端口冲突或内存不足 | 检查端口或重启 Docker |
| 页面加载很慢 | 首次编译 | 等待编译完成或预热 |
